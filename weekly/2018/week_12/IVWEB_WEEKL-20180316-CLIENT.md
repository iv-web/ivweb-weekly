data-cke-saved-src="https://static.oschina.net/uploads/space/2018/0316/000632_E0eq_3703517.jpg" src="https://static.oschina.net/uploads/space/2018/0316/000632_E0eq_3703517.jpg"/></p><ul class=" list-paddingleft-2"><li><p><strong>Flow</strong></p></li></ul><p>Flow是Corda中另一个重要的特性，本质上来说Flow就是一系列复杂的Command指令的编排，用来描述自然世界里涉及多方、多环节、有条件的复杂交易流程。因为Corda的共识是transaction级别参与方范围的，同时transaction的通信都是点对点的，所以flow的设计和实现非常直观和简单。<br/><br/>因为面向金融行业，Corda内置了大量开箱即用的Flow template（在<code>net.corda.flows</code>包），基本涵盖了金融领域主要交易流程。同时Flow支持组合和继承，方便自定义和编排。<br/><br/>Flow的底层实现是基于<code>Quasar</code>的，是JVM平台上实现了actor模型的纤程/轻线程库（Akka也是，但主要面向Scala）。通过Quasar的bytecode注入，可以实现flow的挂起、恢复等调度，这能进一步提高Corda系统的伸缩能力和并发能力。</p><h3><strong>数据存储</strong></h3><ul class=" list-paddingleft-2"><li><p><strong>JPA</strong></p></li><li><p><strong>RDB</strong></p></li></ul><p>Corda的数据存储支持标准JPA规范，可以通过多种ORM库将数据持久化到关系型数据库。目前Corda实现里是内置了一个H2数据库。由于标准的JPA规范，这使得Corda节点可以非常容易得集成/复用金融行业使用广泛（几乎是唯一）的企业级RDB系统，比如DB2或OracleDB。<br/><br/>数据层面这种开放架构，使得企业客户完全能够将Corda的数据和自身业务数据无缝集成。比如通过SQL Join来统一查询，或输入到Hadoop进行离线计算等。</p><h3><strong>网络通信</strong></h3><ul class=" list-paddingleft-2"><li><p><strong>AMQP over TLS</strong></p></li></ul><p>Corda借助ActiveMQ这个流行的消息中间件中的Artemis项目，以AMQP作为其网络通信协议，包括消息结构、序列化格式、加密方式等。</p><p>Artemis本身是个高性能、非阻塞的（基于Netty）支持多种协议的（AMQP、MQTT等）轻量级消息中间件。</p><ul class=" list-paddingleft-2"><li><p><strong>NetworkMapCache</strong></p></li></ul><p>P2P网络部分，Corda通过NetworkMapCache使得每个节点都缓存一份网络拓扑，网络的变化会通知到所有节点。节点本身具备动态发现、注册和认证的能力。</p><p>目前的实现中，是简单得通过seed nodes的方式来初始化整个网络，代码注释里提到未来会集成Paxos或Raft来选举初始网络。</p><h3><strong>CorDapp</strong></h3><ul class=" list-paddingleft-2"><li><p><strong>Plugin机制</strong></p></li><li><p><strong>REST</strong></p></li><li><p><strong>Fatjar</strong></p></li></ul><p>类似Hyperledger Fabric，Corda也采用了plugin机制来支持CorDapp。每个CorDapp需要扩展<code>CordaPluginRegistry</code>这个接口来注册自己，并通过REST API来向外提供服务。</p><p>CorDapp中的逻辑是通过声明或使用flow来实现的，客观上提高了安全性。CorDapp会打包成Fatjar的形式，上传并部署到Corda节点的JVM中。</p><h3><strong>系统集成及其他</strong></h3><p>利用Corda支持的JPA/JDBC，AMQP，REST等等这些数据层面、通信协议以及CorDapp接口，提供了非常开放易于集成的能力。</p><p>Corda节点内置一个web app，用来管理节点状态、网络状况和Dapp，比如交易记录等。</p><p><strong>架构设计</strong></p><p>下图是Corda的逻辑架构</p><p><img alt="" data-cke-saved-src="https://static.oschina.net/uploads/space/2018/0316/000714_vOmC_3703517.jpg" src="https://static.oschina.net/uploads/space/2018/0316/000714_vOmC_3703517.jpg"/></p><p>2017年低，Corda和微软Azure达成合作，将Corda平台搬到Azure云上，推广更易使用的“分布式账本技术即服务”。</p><p><img alt="" height="361" data-cke-saved-src="https://static.oschina.net/uploads/space/2018/0316/000740_OxbC_3703517.jpg" src="https://static.oschina.net/uploads/space/2018/0316/000740_OxbC_3703517.jpg" <pre><code class="language-javascript">button.on('click', (e) => {

    if (event.data.name === 'scroll') {

});
</code></pre>

<h3 id="abstraction-is-key">Abstraction Is Key</h3>

<p>Rather than forcing our developers to handle these conditionals inside their code, we built an abstraction layer between their content and the environment. We call this layer the &lsquo;wrapper.&rsquo;</p>

<p>Instead of querying the DOM or native browser events directly, we can now proxy our request through the <code>wrapper</code> module.</p>

<pre><code class="language-javascript">import wrapper from 'wrapper';
button.on('click', () => {
    wrapper.scrollTo(resultsContainer.offsetTop);
});
</code></pre>

<p>Each platform has its own wrapper implementation conforming to a common interface of wrapper methods. The wrapper wraps itself around our content and handles the complexity for us.</p>

<h3 id="the-wrapper-pattern">The Wrapper Pattern</h3>

<p>Using wrappers results in code that is cleaner, more readable and consistent between projects. It also allows for micro-optimisations over time, as we make incremental improvements to the wrappers to make their methods more performant and accessible. Your project can, therefore, benefit from the experience of many developers.</p>

<p>So, what does a wrapper look like?</p>

<h3 id="wrapper-structure">Wrapper Structure</h3>

<p>Each wrapper essentially comprises three things: a Handlebars template, wrapper JS file, and a SASS file denoting wrapper-specific styling. Additionally, there are build tasks which hook into events exposed by the underlying scaffolding so that each wrapper is responsible for its own pre-compilation and cleanup.</p>

<p>This is a simplified view of the embed wrapper:</p>

<pre><code class="language-javascript">embed-wrapper/
    templates/
        wrapper.hbs
    js/
        wrapper.js
    scss/
        wrapper.scss
</code></pre>

<p>Our underlying scaffolding exposes your main project template as a Handlebars partial, which is consumed by the wrapper. For example, <code>templates/wrapper.hbs</code> might contain:</p>

<pre><code class="language-html">&lt;div class="bbc-news-vj-wrapper--embed"&gt;
    {{&gt;your-application}}
&lt;/div&gt;
</code></pre>

<p><code>scss/wrapper.scss</code> contains wrapper-specific styling that your application code shouldn&rsquo;t need to define itself. The embed wrapper, for example, replicates a lot of BBC News styling inside the iframe.</p>

<p>Finally, <code>js/wrapper.js</code> contains the iframed implementation of the wrapper API, detailed below. It is shipped separately to the project, rather than compiled in with the application code &mdash; we flag <code>wrapper</code> as a global in our Webpack build process. This means that though we deliver our application to multiple platforms, we only compile the code once.</p>

<h4 id="wrapper-api">Wrapper API</h4>

<p>The wrapper API abstracts a number of key browser interactions. Here are the most important ones:</p>

<p><strong><code>scrollTo(int)</code></strong></p>

<p>Scrolls to the given position in the active window. The wrapper will <em>normalise</em> the provided integer before triggering the scroll so that the host page is scrolled to the correct position.</p>

<p><strong><code>getScrollPosition: int</code></strong></p>

<p>Returns the user&rsquo;s current (normalized) scroll position. In the case of the iframe, this means that the scroll position passed to your application is actually <em>negative</em> until the iframe is at the top of the viewport. This is super useful and lets us do things such as animate a component only when it comes into view.</p>

<p><strong><code>onScroll(callback)</code></strong></p>

<p>Provides a hook into the scroll event. In the standalone wrapper, this is essentially hooking into the native scroll event. In the embed wrapper, there will be a slight delay in receiving the scroll event since it is passed via postMessage.</p>

<p><strong><code>viewport: {height: int, width: int}</code></strong></p>

<p>A method to retrieve the viewport height and width (since this is implemented very differently when queried from within an iframe).</p>

<p><strong><code>toggleFullScreen</code></strong></p>

<p>In standalone mode, we hide the BBC menu and footer from view and set a <code>position: fixed</code> on our content. In the News App, we do nothing at all &mdash; the content is already full screen. The complicated one is the iframe, which relies on applying styles both inside and outside the iframe, coordinated via postMessage.</p>

<p><strong><code>markPageAsLoaded</code></strong></p>

<p>Tell the wrapper your content has loaded. This is crucial for our content to work in the News App, which will not attempt to display our content to the user until we explicitly tell the app our content is ready. It also removes the loading spinner on the web versions of our content.</p>

<h4 id="list-of-wrappers">List Of Wrappers</h4>

<p>In the future, we envisage creating additional wrappers for large platforms such as Facebook Instant Articles and Apple News. We have created six wrappers to date:</p>

<p><strong>Standalone Wrapper</strong></p>

<p>The version of our content that should go in standalone pages. Comes bundled with BBC branding.</p>

<p><strong>Embed Wrapper</strong></p>

<p>The iframed version of our content, which is safe to sit inside articles or to syndicate to non-BBC sites, since we retain control over the content.</p>

<p><strong>AMP Wrapper</strong></p>

<p>This is the endpoint which is pulled in as an <code>amp-iframe</code> into AMP pages.</p>

<p><strong>News App Wrapper</strong></p>

<p>Our content must make calls to a proprietary <code>bbcvisualjournalism://</code> protocol.</p>

<p><strong>Core Wrapper</strong></p>

<p>Contains only the HTML &mdash; none of our project&rsquo;s CSS or JavaScript.</p>

<p><strong>JSON Wrapper</strong></p>

<p>A JSON representation of our content, for sharing across BBC products.</p>

<h3 id="wiring-wrappers-up-to-the-platforms">Wiring Wrappers Up To The Platforms</h3>

<p>For our content to appear on the BBC site, we provide journalists with a namespaced path:</p>

<pre class="break-out"><code class="language-markup">/include/[department]/[unique ID], e.g. </code><code class="language-markup" style="font-weight: bold;">/include/visual-journalism/123-quiz</code></pre>

<p>The journalist puts this &ldquo;include path&rdquo; into the CMS, which saves the article structure into the database. All products and services sit downstream of this publishing mechanism. Each platform is responsible for choosing the flavor of content it wants and requesting that content from a proxy server.</p>

<p>Let&rsquo;s take that  from earlier. Here, the include path in the CMS is:</p>

<pre><code class="language-markup">/include/newsspec/15996-trump-tracker/english/index
</code></pre>

<p>The canonical article page knows it wants the &ldquo;embed&rdquo; version of the content, so it appends <code>/embed</code> to the include path:</p>

<pre><code class="language-markup">/include/newsspec/15996-trump-tracker/english/index</code><code class="language-markup" style="font-weight: bold;">/embed</code></pre>

<p>&hellip;before requesting it from the proxy server:</p>

<pre class="break-out"><code class="language-markup">https://news.files.bbci.co.uk/include/newsspec/15996-trump-tracker/english/index/embed
</code></pre>

<p>The AMP page, on the other hand, sees the include path and appends <code>/amp</code>:</p>

<pre><code class="language-markup">/include/newsspec/15996-trump-tracker/english/index</code><code class="language-markup" style="font-weight: bold;">/amp</code></pre>

<p>The AMP renderer does a little magic to render some AMP HTML which references our content, pulling in the <code>/amp</code> version as an iframe:</p>

<pre class="break-out"><code class="language-html">&lt;amp-iframe src="https://news.files.bbci.co.uk/include/newsspec/15996-trump-tracker/english/index/amp" width="640" height="360"&gt;
    &lt;!-- some other AMP elements here --&gt;
&lt;/amp-iframe&gt;
</code></pre>

<p>Every supported platform has its own version of the content:</p>

<pre class="break-out"><code class="language-markup">/include/newsspec/15996-trump-tracker/english/index</code><code class="language-markup" style="font-weight: bold;">/amp</code>

<code>/include/newsspec/15996-trump-tracker/english/index</code><code class="language-markup" style="font-weight: bold;">/core</code>

<code>/include/newsspec/15996-trump-tracker/english/index</code><code class="language-markup" style="font-weight: bold;">/envelope</code>

...and so on</pre>

<p>This solution can scale to incorporate more platform types as they arise.</p>

<h3 id="abstraction-is-hard">Abstraction Is Hard</h3>

<p>Building a &ldquo;write once, deploy anywhere&rdquo; architecture sounds quite idealistic, and it is. For the wrapper architecture to work, we have to be <em>very</em> strict on working within the abstraction. This means we have to fight the temptation to &ldquo;do this hacky thing to make it work in [insert platform name here].&rdquo; We want our content to be completely unaware of the environment it is shipped in &mdash; but this is easier said than done.</p>

<h4 id="features-of-the-platform-are-hard-to-configure-abstractly">Features Of The Platform Are Hard To Configure Abstractly</h4>

<p>Before our abstraction approach, we had complete control over every aspect of our output, including, for example, the markup of our iframe. If we needed to tweak anything on a per-project basis, such as add a <code>title</code> attribute to the iframe for accessibility reasons, we could just edit the markup.</p>

<p>Now that the wrapper markup exists in isolation from the project, the only way of configuring it would be to expose a hook in the scaffold itself. We can do this relatively easily for cross-platform features, but exposing hooks for specific platforms breaks the abstraction. We don&rsquo;t really want to expose an &lsquo;iframe title&rsquo; configuration option that&rsquo;s only used by the one wrapper.</p>

<p>We could name the property more generically, e.g. <code>title</code>, and then use this value as the iframe <code>title</code> attribute. However, it starts to become difficult to keep track of what is used where, and we risk abstracting our configuration to the point of no longer understanding it. By and large, we try to keep our config as lean as possible, only setting properties that have global use.</p>

<h3 id="plans-for-the-future">Plans For The Future</h3>

<p>We hope to develop new wrappers for platforms such as Apple News and Facebook Instant Articles, as well as to offer all new platforms a &lsquo;core&rsquo; version of our content out of the box.</p>

<p>We also hope to get better at progressive enhancement; succeeding in this field means developing defensively. You can never assume all platforms now and in the future will support a given interaction, but <strong>a well-designed project should be able to get its core message across</strong> without falling at the first technical hurdle.</p>

<p>Working within the confines of the wrapper is a bit of a paradigm shift, and feels like a bit of a halfway house in terms of the <em>long-term</em> solution. But until the industry matures onto a cross-platform standard, publishers will be forced to roll out their own solutions, or use tooling such as  for platform-to-platform conversion, or else ignore entire sections of their audience altogether.</p>

<p>It&rsquo;s early days for us, but so far we&rsquo;ve had great success in using the wrapper pattern to build our content once and deliver it to the myriad of platforms our audiences are now using.</p>
